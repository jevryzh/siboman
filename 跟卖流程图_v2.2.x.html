02类目解析 — 4 档置信度

插件采集每个商品时会调 /api/seller/products/category-resolve, 服务端会返回 5 级 fallback 的结果, 综合为以下 4 档置信度:

🟢 高置信 high

服务端找到本店已有商品的 2 个不同 token 都命中 的名字 (例: "кемпинговый" AND "фонарь" 都出现在历史商品名里), 或 SKU 完全匹配本地库。 直接提交, 不需要人工确认。

🟡 中置信 medium

只有 1 个 token 命中, 或同一 type_id 在本店历史归过此类目。 系统自动用, 但用户可以"换一个" — 表格里点链接打开 modal 重选。

🔴 未解析 none

名字 0 命中, type_id 也无历史。返回该店 top-10 高频类目作为 候选, 表格里显示 选类目 (N) 按钮, 用户必点选才能提交。

🔵 已手动选 manual

用户从候选 modal 选了类目, 或 localStorage 自动恢复了之前选过的类目。 自动提交。手动选过的类目会被记住, 下次同 SKU 采集后自动恢复。

键盘快捷键

类目选错了怎么办? 选错不要紧 — Ozon 接受后会在 ListingHistory 显示 imported, 你去 seller.ozon.ru 后台改类目就行。ERP 这边不做"类目合法性"二次校验, Ozon 是 source of truth。

localStorage 记忆机制

用户手动选过的类目会被缓存在 localStorage['zhumeng_category_choice_v1'], key 是 SKU, 最多保留 500 条。下次同 SKU 采集完成后, 服务端给 none 时, 前端自动从这个 cache 恢复 → 状态变 manual, source = history-auto。

04插件版本与安装

插件跟随 ERP 版本号同步。 v2.2.x = 增量改进, 不破坏接口。 升级路径是 chrome://extensions → 卸旧 → 装新 zip:

  1. 下载新版 zip

    访问 http://test.renwz.cn:5178/extension/zhumeng-collector.zip, 保存到本地。

  2. 删除旧版

    打开 chrome://extensions/, 找到"逐梦 Ozon 采集器", 点 移除。

  3. 开发者模式加载

    右上角打开"开发者模式"开关 → 点 加载已解压的扩展程序 → 选刚才下载并解压的 zhumeng-collector 文件夹。

  4. 登录 Ozon Seller

    打开 seller.ozon.ru 标签页并登录, 插件会自动检测 cookie 状态。

  5. 回到 ERP 刷新页面

    在 ERP 的 BatchUpload 页面, 顶部应该显示"🟢 插件已连接"。如果不是, 点 🔄 刷新状态。

版本号一致性检查清单

05常见问题

Q1: 采集后所有商品都显示"⚠ 未解析", 没有候选?

这通常意味着这家店 还没在 ERP 后台同步过任何商品 (app_products 表为空), 或者同步过但 description_category_id 字段全空。解决: 去 app_stores 配 client_id + api_key → 触发首次 Sync-All → 等 sync 完成 → 再试采集。

Q2: 选了类目还是上传失败, Ozon 返回 400 "category not found"?

可能候选类是 这家店不存在 的类目 (树缓存过期或 storeId 错)。刷新类目树: 顶部 🔄 刷新状态 → 后台会调用 /v1/description-category/tree 重建缓存。如果还失败, 去 seller.ozon.ru 后台选。

Q3: 我手动选过 1 次类目, 下次同 SKU 应该自动恢复, 但没生效?

检查 3 个条件: ① SKU 完全相同 (含前后 0) ② 该行 _category_resolved.confidence === 'none' (不是 manual) ③ localStorage 没被浏览器清掉 (隐身模式 / 浏览器 data 清理会丢)。如果都满足, 看控制台有没有 💾 N 个商品自动恢复 的日志。

Q4: type_id 我知道是 95120, 但插件说 0, 没拿到?

Ozon 公开页面不展示 type_id, 只有 seller API 才返回。插件会通过 OPI 辅源拉同店同款的 type_id — 如果你店 完全没卖过这个 type_id 的商品, 就拿不到。解决: 选个相似商品的 type_id (modal 的 type_id 兜底会自动按店铺历史推荐)。

Q5: 4 个店(LuckyDay / Polarwind / Superstar / Three Latte)能同时跑吗?

可以。BatchUpload 顶部 目标店铺 多选 (Ctrl/Cmd 点), 服务端会扇出到每个店, 每个店独立调它的 client_id + api_key。但 同一 SKU 在多个店不能完全同图同标题, Ozon 会判定重复商品。建议: 同店复用, 跨店至少换 1-2 张图、改改标题。

Q6: 上架后我看哪个页面?

顶部菜单 → 📜 历史记录 或直接 #/listing-history。每 30 秒自动刷新, 显示 Ozon 接受的 status (imported / processing / failed)。Ozon 异步审核可能延迟 5-30 分钟, 看到 processing 别慌。

反馈入口: 发现 bug / 想加功能, 直接在团队群里 ping 逐梦 ERP 助手, 把控制台日志 (F12 → Console) + 截屏发出来即可。版本号是判断"是不是最新代码"的最快方式, 永远附带插件版本号 (右上角 StoreManagement 卡片可查)。
