## 0. 写在最前（对用户诉求的回应）

> 用户原话："改一个小需求就重构。重构完功能全部不完整，功能混乱，流程混乱，连项目经理都拉跨。"

**我承认这是产品侧的问题**。之前的做法有 3 个错误：

1. **没先定义"完整"的边界** → 每次都是"新做一个小功能"，导致 P0 之外的旧功能被弃、被砍、被遗忘
2. **没定义组件规范** → 每个页面各写一套表格、各写一套弹窗，改一个动一片
3. **没做回归清单** → 上线新版本时没人检查"旧版能做的现在还能做吗"

**这份文档解决 3 件事**：
- **§1-§3** 完整功能清单 —— 明确"0.2.0 上线时必须能干什么"，杜绝功能缺失
- **§4** 组件规范 —— 明确"所有列表页/表单页必须长什么样"，杜绝改一片崩一片
- **§5** 回归检查清单 —— 上线前逐条勾，勾不完不上线

#### 2.1.3 批量操作栏（勾选后从顶部滑入）

```
┌─────────────────────────────────────────────────────────────────┐
│ 已选 20 个商品                                                   │
│ [批量修改售价] [批量修改库存] [批量体检] [批量同步Ozon] [批量下架]│
└─────────────────────────────────────────────────────────────────┘
```

- **批量修改售价** → 弹窗：`[加价 ▼] [10 %] [执行]`  支持 `加价%` / `加价额` / `设为固定值` / `按利润率反推` 4 种模式
- **批量修改库存** → 弹窗：`[设为 ▼] [100] [执行]`  支持 `设为固定值` / `增加` / `减少`
- **批量体检** → 12 条规则跑一遍，结果生成 Excel 报告下载
- **批量同步 Ozon** → 循环调 `/v1/product/import`，进度条 X/N
- **批量下架** → 二次确认（不可撤销），循环调 Ozon

| ID | Given / When / Then |
|---|---------------------|
| AC-PL01 | Given 商品列表；When 双击售价单元格；Then 变输入框，Esc 取消，Enter 保存并调 Ozon 增量更新 API |
| AC-PL02 | Given 勾选 3 个商品；When 点批量修改售价选"加价 10%"；Then 弹进度条 3/3，成功后列表自动刷新 |
| AC-PL03 | Given error 状态商品；When 点状态徽章；Then 弹窗显示 Ozon 返回的完整俄文错误原文 + 中文翻译 |
| AC-PL04 | Given 库存=0 的商品；Then 库存单元格红色高亮，售价单元格底色浅红 |
| AC-PL05 | Given 点击[编辑]；Then 打开右侧抽屉（复用 v1 PRD §2.1 的采集箱编辑抽屉，字段全部预填） |

## 4. 【核心】组件规范 —— 解决"改一片崩全站"

> **这一节是本次文档的灵魂**。用户反馈的"改一点崩全站"根因是**每个页面各写一套**。0.2.0 起，所有列表页/抽屉/弹窗必须走**统一组件**。

### 4.1 强制统一的 5 个基础组件

Coder 必须先做完这 5 个组件，再开始做业务页面。**禁止**每个页面自己拼 DOM。

#### C1. `<AppTable>` 统一列表组件

必须支持的能力（不管业务是商品/订单/库存都用这一个）：

```js
props = {
  columns: [
    { key, title, width, render, editable, sortable }
  ],
  dataSource: [],
  loading: false,
  selection: { type: 'checkbox', onChange },
  pagination: { mode: 'cursor', pageSize: 50, onLoadMore },
  emptyText: '暂无数据',
  onRowClick,
  onCellEdit: (row, key, newValue) => Promise
}
```

**约束**：
- 所有列表页只允许用 `<AppTable>`，不许再手写 `<table>`
- 行内编辑（editable: true）走统一 `onCellEdit` 回调
- 空态、loading、错误态都走内置

#### C2. `<AppDrawer>` 统一抽屉

商品编辑、采集箱编辑、订单详情全用这一个组件。

```js
props = {
  title,
  width: 800,
  visible,
  onClose,
  onSave,           // 内部触发保存
  loading,
  dirty: false,     // 有未保存修改时 close 弹拦截
  footer: {
    progress: '1/17 · 6%',  // 顶栏/底栏进度对标 MyERP
    leftBtn: '保存草稿',
    rightBtn: '上架 Ozon',
    disabled: true
  }
}
slots = { default }   // 抽屉里放什么表单是业务决定
```

#### C3. `<AppFormField>` 统一表单字段

所有输入字段必须走这一个（避免 5 个页面 5 种下拉长得不一样）。

```js
type = 'text' | 'number' | 'select' | 'autocomplete' | 'cascader' | 'multiselect-search' | 'switch' | 'richtext' | 'image-upload'

props = {
  label,
  required,
  type,
  value,
  onChange,
  options,          // select 的选项 or 异步 loader
  helpText,
  errorMsg,
  suffix,           // 右侧按钮，如 [fx 利润反推]
}
```

- 类目字段 = `type: 'cascader'`
- 品牌字段 = `type: 'autocomplete'`
- Ozon 属性字段 = 后端返回 `{type: 'multiselect-search', options: [...]}` 前端不用改代码

#### C4. `<AppBatchBar>` 统一批量操作栏

勾选后滑入的顶部横条。

```js
props = {
  selectedCount,
  actions: [
    { key, label, icon, dangerous, onClick }
  ],
  onCancel
}
```

- dangerous 的 action（如"批量下架"）自动加二次确认弹窗
- 所有异步 action 自动包裹进度条 X/N

#### C5. `<AppConfirmDialog>` 统一确认弹窗

不可撤销操作（批量备货、下架、删除）必须走这一个。

```js
type = 'danger' | 'warning' | 'info'
props = { title, content, dangerous: true, confirmText: '确认执行', onConfirm }
```

### 4.2 组件-页面 归属矩阵

| 页面 | AppTable | AppDrawer | AppFormField | AppBatchBar | AppConfirmDialog |
|-----|---------|----------|-------------|-------------|-----------------|
| 商品列表 | ✅ 主表格 | ✅ 编辑抽屉 | ✅ 抽屉内所有字段 | ✅ 批量操作 | ✅ 批量下架 |
| 采集箱 | ✅ | ✅（复用同一个） | ✅ | ✅ | ✅ |
| 库存管理 | ✅ | 🟡 补货弹窗 | ✅ | ✅ 批量改库存 | — |
| 订单列表 | ✅ | ✅ 详情弹窗 | 🟡 备注 | ✅ 批量备货/面单 | ✅ 批量备货 |
| 消息中心 | ✅ | — | — | — | — |

**关键规则**：**同一个抽屉组件被 3 个页面复用**（商品编辑 / 采集编辑 / 后续新页面），改一次全站生效。

### 4.3 技术选型建议（前端）

> 用户已表达"没指定框架，但原生 JS 已经改不动了"。

**强烈推荐**：**Vue 3 + Element Plus**

理由：
1. Element Plus 自带的组件 90% 匹配 §4.1 的需求：
   - `el-table` = C1
   - `el-drawer` = C2
   - `el-cascader / el-autocomplete / el-select-v2` = C3
   - `el-message-box` = C5
2. 无需 build 工具（用 `<script type="module">` 引 CDN 即可），学习成本 3-5 天
3. 中文文档全，遇到问题 Google 就有答案
4. 迁移路径可以渐进：新页面用 Vue，老页面暂不动

**迁移路径**：
1. week 1：搭 Vue 3 + Element Plus 脚手架，把 §4.1 的 5 个组件封装好
2. week 2-3：**新做的 P0 页面全用 Vue 写**（商品列表、采集编辑、订单列表等）
3. week 4：老的原生 JS 页面按需重构或保留（不动就不动，不要瞎动）

这一节是**产品视角的建议**，最终框架选型由 Coder 定。但**不能不换**——继续用原生 JS + `innerHTML` 就是继续"改一片崩全站"。

### 4.4 后端接口规范（配合前端解耦）

所有列表接口必须走统一响应格式：

```json
{
  "success": true,
  "items": [...],
  "pagination": {
    "last_id": "cursor",
    "has_more": true,
    "total": 1234
  },
  "error": null
}
```

所有写操作必须返回：

```json
{
  "success": true,
  "affected": 3,
  "failed_items": [
    { "id": "xxx", "reason": "..." }
  ]
}
```

**理由**：前端 `<AppBatchBar>` 的进度条只需要认这一个格式，不用每个批量操作都改前端。
