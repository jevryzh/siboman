## 1. 业务背景与详尽现状分析

### 1.1 项目技术现状深度审计
根据 [HANDOFF.md](HANDOFF.md) §10.3 以及对 `server.js` (L630) 和 `public/app.js` (L877) 的源码审计，目前的库存管理模块存在以下核心技术瓶颈：
- **同步机制简陋**：目前的库存更新是“即时且同步”的。用户在前端输入数值并点击提交后，系统立即通过 `POST /v2/products/stocks` (server.js:655) 向 Ozon 发送请求。这种模式在需要批量调整几百个 SKU 的库存时，效率极低且极易因为网络闪断导致部分更新失败。
- **无状态持久化**：目前系统没有“库存草稿”概念。如果运营人员填写了 50 个 SKU 的库存，但由于临时有事关闭了浏览器，之前填写的数据将全部丢失，必须重新核对并输入。
- **预警机制缺失**：目前的列表仅展示当前数值。卖家无法一眼识别出哪些商品已经进入“缺货”或“低库存”警戒区。对于动辄几百个 SKU 的店铺，人工核对库存的压力巨大。
- **仓库逻辑硬编码**：当前获取仓库列表的方式（`server.js:606`）是从 FBS 订单列表中提取。如果是一个全新的店铺，还没有出过单，系统将无法正确显示该店铺的仓库，导致无法更新库存。

### 1.2 核心业务痛点 (Pain Points)
1.  **超卖风险极高**：由于缺乏“低库存”提醒，卖家往往在 Ozon 订单生成后才发现 1688 已经断货，导致无法发货被平台重罚。
2.  **操作效率低下**：不支持 Excel 导入。对于拥有本地进销存系统的成熟卖家，必须手动在两个系统间搬运数据。
3.  **多仓管理混乱**：Ozon 允许卖家配置多个仓库（如莫斯科仓、哈巴罗夫斯克仓），目前的 UI 对多仓显示支持不佳，容易造成错填。
4.  **数据安全感差**：点击提交后，如果其中一个 SKU 报错，由于缺乏日志记录，卖家很难定位是哪个 SKU 没改成功。

## 2. 需求目标与非目标 (Detailed Scope)

### 2.1 本次迭代目标
1.  **引入“库存草稿箱”**：实现本地缓存机制。用户修改后的数值会自动存入 `app_stock_drafts`，支持“存为草稿”和“批量提交”。
2.  **多维度库存看板**：在列表上方增加四个核心指标卡：商品总数、缺货数、低库存数、暂存待提交数。
3.  **动态低库存预警**：支持用户自定义“低库存阈值”（默认为 5）。当库存低于该值时，行背景及状态标签自动变红。
4.  **支持 Excel 批量导入**：提供标准的 Excel 模板。卖家通过上传 CSV/XLSX，系统自动匹配 Offer ID 并预填草稿。
5.  **官方仓库直连**：废弃“从订单提取”的方案，改用 Ozon 官方 API（待确认 Client 权限）或增强型同步逻辑获取全量仓库。

### 2.2 非目标
1.  **自动补货采购**：本阶段仅负责同步。自动向 1688 下单补货功能由 V3 版本选品中台实现。
2.  **FBO 入仓管理**：本模块聚焦于 FBS (Seller Fulfilled) 模式。

## 5. 后端 API 详细设计 (Deep Technical)

### 5.1 获取库存草稿列表 (GET /api/seller/stocks/drafts)
- **用途**：获取本地暂存且未同步到 Ozon 的变更值。
- **Response 200**：
```json
{
  "items": [
    {
      "offer_id": "ER-10239",
      "warehouse_id": 123456,
      "diff_value": 15,
      "updated_at": "2026-07-02T12:00:00Z"
    }
  ]
}
```

### 5.2 保存暂存草稿 (POST /api/seller/stocks/save-draft)
- **Request Body**：
```json
{
  "stocks": [
    { "offer_id": "SKU-01", "warehouse_id": 123, "diff": 10 },
    { "offer_id": "SKU-02", "warehouse_id": 123, "diff": -2 }
  ]
}
```

### 5.3 批量提交到 Ozon (POST /api/seller/products/stocks/bulk)
- **逻辑说明**：
  1. 后端读取 `app_stock_drafts` 中勾选的项。
  2. 按照 Ozon API `/v2/products/stocks` 的 100 个/批限制进行拆分。
  3. 依次调用 Ozon 接口。
  4. 成功后，物理删除本地对应的 `app_stock_drafts` 记录。

## 6. 数据模型变更 (PostgreSQL DDL)

### 6.1 app_stock_drafts 表设计 (库存草稿箱)
```sql
CREATE TABLE IF NOT EXISTS app_stock_drafts (
  id BIGSERIAL PRIMARY KEY,
  user_id BIGINT NOT NULL REFERENCES app_users(id),
  
  -- 唯一定位 SKU + 仓库
  offer_id VARCHAR(100) NOT NULL,
  warehouse_id BIGINT NOT NULL,
  
  -- 变更详情
  current_ozon_stock INTEGER DEFAULT 0, -- 暂存时快照，用于前端对比
  new_diff_value INTEGER NOT NULL, -- 增量值 (如 +5, -2)
  
  -- 审计与同步
  is_processing BOOLEAN DEFAULT FALSE,
  last_error TEXT,
  
  -- 时间戳
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  
  -- 约束：一个用户下的一个 SKU 的一个仓库只能有一条草稿
  UNIQUE(user_id, offer_id, warehouse_id)
);

CREATE INDEX idx_stock_draft_user ON app_stock_drafts(user_id);
COMMENT ON TABLE app_stock_drafts IS '库存变更本地暂存草稿表';
```

## 7. 业务逻辑流程图 (Process Flow)

1.  **输入阶段**: 用户在列表 `变更值` 列输入 `+10`。
2.  **暂存阶段**: 输入框 `blur` 事件触发，调用 `save-draft` API。前端 `变更预计` 立即更新为 `当前 + 10`。
3.  **筛选阶段**: 用户点击 `暂存草稿` Tab，检查所有待提交的项。
4.  **同步阶段**: 用户点击 `批量提交到 Ozon`。
5.  **反馈阶段**: 后端返回结果。若 Ozon 提示 `warehouse_id invalid`，则本地草稿保留，并在行内标记红色报错。

## 8. Excel 导入规约 (Import Specs)

- **支持格式**: `.csv`, `.xlsx`
- **必填列**: `Offer ID` (货号), `Quantity` (变更后的总值) 或 `Diff` (增减值)。
- **逻辑**: 系统自动根据 `Offer ID` 匹配本地商品库。若匹配失败，在导入报告中列出“未找到的 SKU”。

## 9. 验收标准 (Acceptance Criteria - 15 Items)

- **AC-01 [草稿持久化]**: 输入数值并刷新页面，数值必须依然存在。 → US-02
- **AC-02 [状态着色]**: 当 `stock < threshold` 时，行背景必须变色，且状态列显示“低库存”。 → US-01
- **AC-03 [批量提交]**: 选中 50 个商品点击提交，必须在 3 秒内显示提交结果及成功/失败比例。 → US-02
- **AC-04 [仓库准确性]**: 必须能显示 Ozon Seller 所有的活跃仓库（基于 `/v3/posting/fbs/list` 提取）。 → US-04
- **AC-05 [Excel 导入]**: 导入 100 行 Excel，匹配成功的项应立即填充到草稿中，并在界面提示“成功导入 80 条”。 → US-04
- **AC-06 [数值安全]**: 不允许提交会导致库存变为负数的变更值。
- **AC-07 [指标同步]**: 顶部指标卡（如“低库存数”）必须随搜索过滤和草稿修改实时更新。 → US-03
- **AC-08 [操作日志]**: 每一笔成功的 Ozon 同步操作必须在 `app_jobs` 或专用日志表中可查。 → US-05
- **AC-09 [快速归档]**: 支持在库存页直接一键下架（库存设为 0）。
- **AC-10 [多店支持]**: 如果账号绑定多店，仓库下拉框应根据店铺 ID 动态加载。
- **AC-11 [导出预览]**: “预补货单”导出应包含 1688 链接及最近 7 天销量。 → US-03
- **AC-12 [冲突处理]**: 若 Ozon 端库存已变动，提交前应弹窗提示“实时值已变化，是否继续？”。
- **AC-13 [错误定位]**: 提交失败的 SKU 应精准标红，并气泡显示 Ozon 原始错误代码。
- **AC-14 [性能达标]**: 分页 50 条渲染时间 < 600ms。
- **AC-15 [忽略操作]**: 支持一键清空所有本地草稿。

## 11. 风险与回滚方案 (Risk & Rollback)

### 11.1 主要风险
- **数据不一致**: 本地草稿与 Ozon 实时值存在时间差。对策：提交前强制刷新。
- **Excel 格式错误**: 用户上传了错误的 SKU。对策：提供详尽的错误导入报告。

### 11.2 回滚计划
1.  还原 `public/app.js` 的 `renderProductStock`。
2.  `DROP TABLE app_stock_drafts;`。

## 12. 技术 FAQ (Developer FAQ)
- **Q**: 为什么不直接在前端存 LocalStorage？
- **A**: 采集端和多台电脑协作时需要 DB 持久化，确保数据同步。
- **Q**: Ozon 支持增量同步吗？
- **A**: 不支持，`/v2/products/stocks` 必须传全量目标值。

## 附录 A：库存管理深度实施细节

### A.1 批量同步的重试与幂等逻辑
如果调用 Ozon 接口时网络超时：
1.  将该批次标记为 `retry_pending`。
2.  30 秒后重试。
3.  重试 3 次后若仍失败，标记为 `failed` 并提示用户。

### A.2 Excel 导入的具体字段映射 (Schema)
| 字段名 | 含义 | 是否必填 |
| :--- | :--- | :--- |
| `offer_id` | 货号 | 是 |
| `stock` | 目标库存 | 是 (与 diff 二选一) |
| `diff` | 变更量 | 否 |

### A.3 前端实时预计库存算法 (Preview)
```javascript
function calcPreview(current, diffInput) {
    const d = parseInt(diffInput) || 0;
    return Math.max(0, current + d);
}
```
