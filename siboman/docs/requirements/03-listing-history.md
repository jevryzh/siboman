## 1. 业务背景与详尽现状分析

### 1.1 现状描述
目前逐梦 ERP 的“上架”功能实现位于 `server.js` 的第 708 行，端点为 `/api/seller/products/import`。当前逻辑是同步调用 Ozon Seller API 的 `/v3/products/import` 接口。
- **技术局限**：Ozon 的商品创建是一个典型的异步流程。后端调用 `/v3/products/import` 后，Ozon 会立即返回一个 `task_id`，但这并不代表商品已经创建成功。
- **用户感知缺失**：目前系统在拿到 `task_id` 后就直接向前端返回了成功。用户无法得知该异步任务在 Ozon 内部是否执行成功。如果图片格式不对或属性校验失败，用户必须在几分钟后登录 Ozon Seller Center 才能看到报错。
- **无历史追溯**：目前系统不记录过去的上传操作。如果运营人员不记得自己上午提交了哪些商品，只能去 Ozon 后台人肉核对。

### 1.2 核心痛点 (Pain Points)
1.  **黑盒任务感**：上架点击后，页面转圈结束即认为“已完成”，但实际 Ozon 端可能在 5 分钟后标记为失败。用户无法闭环处理。
2.  **报错信息难以理解**：Ozon API 的异步报错通常是原始的俄语 JSON 数组（例如：`"attribute_id": 85, "error": "value is required"`）。普通运营人员无法直接看懂并进行修复。
3.  **成功率无法统计**：缺乏数据支持，无法得知本店铺上架的“首通率”。

## 2. 需求目标与非目标 (Detailed Scope)

### 2.1 本次迭代目标
1.  **全量记录异步任务**：建立 `app_listing_history` 表，完整记录每一次提交的 Payload 和返回的 `task_id`。
2.  **建立轮询监控机制**：后台或前端定时调用 Ozon `/v1/product/import/info`，自动更新任务状态。
3.  **报错语义化翻译**：将 Ozon 的原始错误代码映射为中文修复建议。
4.  **提供任务重试入口**：针对失败的任务，允许用户点击“一键重试”，自动带入历史 Payload 跳转到上架页。
5.  **上架效能看板**：展示今日上品数、累计批次、当前处理中任务数。

### 2.2 非目标
1.  **定时预约上架**：本阶段仍以即时提交为主。
2.  **批量修改历史记录**：历史记录作为审计日志，不支持手动修改其中的数据内容。

## 5. 后端 API 详细设计

### 5.1 修改上架提交端点 (POST /api/seller/products/import)
- **改动点**：
    1. 调用 Ozon 返回 `task_id`。
    2. 异步在 `app_listing_history` 插入记录。
    3. 返回给前端 `task_id` 和本地记录 ID。

### 5.2 获取上架历史 (GET /api/seller/import/history)
- **Request Params**：`status`, `search`, `page`, `limit`。
- **Response 200**：
```json
{
  "items": [
    {
      "id": 1001,
      "task_id": 123456789,
      "offer_id": "SKU-998",
      "status": "failed",
      "error_summary": "Attribute 85 is mandatory",
      "created_at": "..."
    }
  ],
  "total": 600
}
```

### 5.3 任务进度同步端点 (POST /api/seller/import/sync-task)
- **用途**：强制单次同步 Ozon 任务状态。
- **Logic**: 调用 Ozon `/v1/product/import/info`。

## 6. 数据模型变更 (PostgreSQL DDL)

### 6.1 app_listing_history 表设计
```sql
CREATE TABLE IF NOT EXISTS app_listing_history (
  id BIGSERIAL PRIMARY KEY,
  user_id BIGINT NOT NULL REFERENCES app_users(id),
  task_id BIGINT UNIQUE NOT NULL,
  
  -- 商品基础资料快照
  offer_id VARCHAR(100),
  product_name TEXT,
  main_image TEXT,
  price_rub DECIMAL(12, 2),
  
  -- 状态追踪
  status VARCHAR(20) DEFAULT 'processing', -- processing, imported, failed
  errors_json JSONB, -- 存储 Ozon 返回的完整错误数组
  raw_payload JSONB, -- 存储提交给 Ozon 的原始 Request Body
  
  -- 统计与元数据
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_listing_history_task ON app_listing_history(task_id);
CREATE INDEX idx_listing_history_user ON app_listing_history(user_id, status);

COMMENT ON TABLE app_listing_history IS '商品上架异步任务审计表';
```

## 7. Ozon 错误代码映射引擎 (Audit Engine)

为了将机器语言翻译成人类语言，系统内置以下映射表：
- `error: value is required` & `attribute_id: 85` -> **建议：请填写“品牌”属性。**
- `error: image_resolution_low` -> **建议：图片分辨率需 >= 200px。**
- `error: download_failed` -> **建议：图片 URL 无法访问，请检查货源图链接。**
- `error: duplicate_offer_id` -> **建议：货号（SKU）已在店铺中存在，请修改。**

## 8. 验收标准 (Acceptance Criteria - 12 Items)

- **AC-01 [任务入库]**: 每次上架操作必须产生一条 `app_listing_history` 记录，`task_id` 必须准确。 → US-01
- **AC-02 [状态同步]**: 针对“处理中”的任务，前端页面打开时应至少触发一次后端同步，成功后状态即时更新。 → US-01
- **AC-03 [错误展示]**: 点击失败记录的“查看详情”，弹窗必须展示具体的字段报错（如 Attribute 85）及其对应的中文翻译。 → US-02
- **AC-04 [一键重试]**: 点击“重试”应将 `raw_payload` 注入 `state._pendingUploadData` 并跳转到上架页。 → US-04
- **AC-05 [分页查询]**: 每页 50 条，搜索 SKU 或任务 ID 应在 1s 内返回结果。 → US-05
- **AC-06 [统计看板]**: 顶部统计栏的“成功率”计算公式：`(成功批次 / 总批次) * 100%`。 → US-03
- **AC-07 [数据完整性]**: 历史记录中必须保存商品主图缩略图，方便视觉确认。 → US-01
- **AC-08 [自动清理]**: 超过 90 天的历史记录应支持自动归档或后台物理清理。
- **AC-09 [多账号隔离]**: 用户仅能看到自己名下上架的历史记录。
- **AC-10 [空态反馈]**: 暂无记录时展示引导上架的占位图。
- **AC-11 [导出 CSV]**: 导出文件应包含 Ozon 返回的错误原始文案。 → US-05
- **AC-12 [轮询控制]**: 前端应提供手动刷新按钮，并显示“最后刷新时间”。

## 10. 风险与回滚
- **数据保留期**: Ozon API 仅保留 48 小时内的任务状态。
- **对策**: 本地表持久化最终状态，仅对 `processing` 状态的任务进行 API 轮询。

## 附录 A：上架记录深度实施细节

### A.1 Ozon 异步状态代码映射 (Status Map)
- `processing`: 系统正在排队等待。
- `moderating`: Ozon 正在进行合规性检测。
- `failed`: 发现严重错误，资料被拒绝。
- `imported`: 商品已成功进入库存。

### A.2 轮询 Job (Cron) 详细逻辑
每隔 15 分钟，后台进程查询所有 `status = 'processing'` 的记录。
批量调用 `/v1/product/import/info`。
如果返回中有 `error` 数组，则更新本地记录并发送站内信。

### A.3 错误诊断引擎实现 (JS)
```javascript
function diagnoseError(code) {
    const map = { 85: "品牌必填", 102: "图片太小" };
    return map[code] || "未知错误";
}
```
