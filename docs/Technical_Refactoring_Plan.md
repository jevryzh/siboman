# siboman 技术重构与业务逻辑修复方案

## 1. 现状分析与架构目标
**现状问题**：
- 前端使用原生 JS 直接操作 `innerHTML`，代码高度耦合，状态管理混乱，改动极易引发雪崩式崩溃。
- 业务流程依赖跳转 Ozon 后台，缺乏内部闭环编辑能力。
- 订单发货流程繁琐，缺乏批量自动化处理能力。

**架构目标**：
- **引入 Vue 3 + Element Plus + Vite**：建立组件化开发体系，通过响应式数据驱动 UI，隔离业务逻辑与 DOM 操作。
- **业务闭环**：实现采集箱与商品列表的“ERP 内直接编辑”抽屉，对接 Ozon API 实现数据双向同步。
- **流程自动化**：基于 `pdf-lib` 实现批量面单合并，基于分布式任务队列实现批量备货。

## 2. 模块划分与重构路径

### 2.1 工程化基础设施 (Day 1)
- **Vite 迁移**：配置 Vite 开发环境，支持 SFC (Single File Components)。
- **Axios 封装**：建立 API 抽象层，统一处理 Ozon 认证与错误提示。
- **Pinia 状态管理**：存储全局汇率 (RUB/CNY)、店铺配置和用户信息。

### 2.2 采集箱与编辑器重构 (Day 2-4)
- **EditDrawer 组件**：封装核心编辑逻辑。
    - **el-cascader 级联**：基于 Ozon 类目树构建带 zh-search 属性的级联选择器。
    - **动态属性渲染引擎**：根据类目返回的 `attribute_id` 自动渲染表单项。
    - **利润计算器 (fx)**：实现进价 (CNY) -> 建议售价 (RUB) 的实时换算逻辑。
- **API 升级**：
    - `POST /api/collect/publish`：将编辑后的数据推送至 Ozon `/v3/product/import`。
    - `PUT /api/collect/draft`：保存本地草稿数据至 JSONB 字段。

### 2.3 订单流优化 (Day 5-6)
- **列表精简**：使用 `el-table` 重构订单页，实现“倒计时”动态高亮。
- **批量合并面单**：
    - 后端循环调用 `callOzonSellerAPI('/v2/posting/fbs/package-label')`。
    - 使用 `pdf-lib` 对返回的 PDF Buffer 进行字节流合并。
- **批量备货逻辑**：异步任务队列执行 `/v3/posting/fbs/ship-package`，前端展示实时执行进度条。

### 2.4 AI 逻辑修正 (Day 7)
- **中文解析器**：在 MiniMax 请求中增加中文前置处理逻辑，通过 Prompt 约束 AI 优先提取 `Selling Points`。
- **提示词模板**：从单纯的“翻译标题”切换为“语义重组”，生成符合 Ozon SEO 规范的俄语标题。

## 3. 自测检查清单与稳定性保障

### 3.1 冒烟测试检查点
- **类目切换**：切换不同类目，动态属性表单是否能够清空并加载对应字段？
- **数据同步**：在 ERP 编辑价格后同步，Ozon 后台的 `marketing_price` 是否同步变更？
- **合并下载**：勾选 10 个订单，下载的 PDF 是否包含 10 页且格式正确？
- **异常捕获**：当 Ozon API 报错（如类目冲突）时，前端是否能弹出清晰的错误原因？

### 3.2 冒烟测试覆盖方案
1. **构建校验**：CI 流程中必须通过 `vue-tsc` 类型检查。
2. **逻辑隔离**：严禁在 Vue 组件中直接调用全局 DOM 元素，所有交互必须通过 `props` 和 `emit`。
3. **数据校验**：提交上架前增加“内容体检”步骤，对比字段长度、图片格式、尺寸范围等 Ozon 硬性规则。
