## 0. 用户核心诉求（先对齐）

引用用户原话，翻译成开发能落地的话：

| 用户原话 | 翻译成需求 |
|--------|----------|
| "前端我感觉很混乱，改动一点就全部都用不了" | 现有前端（原生 JS + 手写 DOM）已到极限，**必须引入组件化框架**，见 §5 技术建议 |
| "商品列表要在 ERP 内直接编辑" | 商品列表行「编辑」按钮打开 ERP 内的编辑抽屉/弹窗，**不再跳转 Ozon 后台** |
| "采集箱先编辑再上架，且智能填写用下拉框" | 采集箱新增「编辑」入口（当前只有「送入上架」），字段全部用**下拉/级联/搜索**替代文本输入 |
| "订单列表要精简字段，实现发货流" | 订单列表精简到 5 列以内，批量发货 / 批量打面单必须做 |
| "开发认错没用，事后没用" | 需求侧要给出**可对照的字段级 AC**，让开发不能凭感觉写；测试有明确清单 |

## 2. siboman 三大模块字段级 PRD

### 2.1 【模块 A】采集箱编辑弹窗（Priority 0，MVP 第一步）

#### 2.1.1 触发入口

- 采集箱列表每行新增「**编辑**」按钮（放在「送入上架」左侧）
- 点击后打开**抽屉式面板**（右侧滑出，宽度 800px，非全屏 Modal，可以边看列表边编辑）
- 空的采集商品也可编辑（用于手动新建 SKU）

#### 2.1.2 字段清单（Coder 严格按此实现）

### 2.2 【模块 B】商品列表页 · 行内编辑（Priority 0）

#### 2.2.1 现状 → 目标

- **现状**：商品列表每行只能"看"，没有编辑按钮，改字段要跳去 Ozon 官方后台
- **目标**：每行「编辑」按钮 → 打开与 A 相同的抽屉（复用同一个组件）；已上架商品字段可修改，保存时调 Ozon `/v1/product/import` 增量更新

#### 2.2.2 字段清单

商品列表编辑抽屉的字段与采集箱编辑抽屉**共用一套组件**，差异：

| 差异点 | 采集箱 (A) | 商品列表 (B) |
|-------|----------|-----------|
| 数据源 | collect_items 表 | app_products 表（Ozon 缓存） |
| offer_id | 可编辑（未上架前） | **只读**（已生成） |
| 保存动作 | 保存草稿 / 上架到 Ozon | 保存到 ERP 缓存 / 同步到 Ozon |
| Ozon API | `/v3/product/import` (create) | `/v1/product/import` (update) |

#### 2.2.3 列表页字段（列）

| # | 列 | 内容 | 交互 |
|---|-----|-----|-----|
| B01 | 图 | 主图缩略图 40x40 | 悬停放大 400x400 |
| B02 | offer_id / SKU | 双行 | 点击复制 offer_id → toast |
| B03 | 俄文标题 | 单行，超出省略 | 悬停显示 tooltip 全文 |
| B04 | 状态徽章 | selling / ready / error / archived | 点击 error 弹出 Ozon 报错原文 |
| B05 | 售价 RUB / 利润率% | 双行，利润率< 10% 显示红色 | 点击进入编辑抽屉 A12 |
| B06 | 库存 | 数字 | 行内直接编辑（inplace） |
| B07 | 操作 | [编辑] [同步] [复制] [下架] | [编辑] = 打开 §2.2 抽屉 |

#### 2.2.4 关键 AC

| ID | Given / When / Then |
|---|---------------------|
| **AC-B01** | Given 商品列表页；When 点击 B07「编辑」；Then 抽屉打开，字段全部预填当前 Ozon 缓存值 |
| **AC-B02** | Given 抽屉内修改了售价；When 点击「同步到 Ozon」；Then 调 Ozon 增量更新 API，返回 task_id 存入 app_jobs 表 |
| **AC-B03** | Given 库存字段；When 用户直接在列表行点击库存数字；Then 数字变为可编辑输入框，回车保存，同步 Ozon |
| **AC-B04** | Given 用户改了字段但未保存；When 点击浏览器返回；Then 弹出「未保存的修改会丢失，是否继续？」拦截 |

## 3. MVP 必做清单（第一步只做这些）

如果时间只够做**一件事**，那就是 §2.1 采集箱编辑抽屉。以下是最小闭环：

### 3.1 P0-1（3 天）：采集箱编辑抽屉基础版

必须包含字段：**A01, A02, A03, A04, A07, A08, A09（仅必填属性）, A10, A11, A12, A14, A15**

## 4. 技术架构建议（针对"前端一改就崩"）

> 这一节是产品视角的**架构建议**，最终由 Coder 定夺。

### 4.1 根因诊断

- **现状**：`public/app.js` 一个文件，用原生 JS + `innerHTML` 拼字符串渲染 DOM，无组件化、无状态管理、无类型系统
- **表现**：改一个字段要动 5 处代码（HTML 拼接 / 事件绑定 / 数据获取 / 校验 / 提交），漏一处就崩
- **本质**：**架构债**，不是"仔细看"能解决的

### 4.2 建议方案（三选一，按侵入性从低到高）

| 方案 | 侵入性 | 学习成本 | 推荐度 |
|-----|-------|--------|--------|
| A. 保留原生 JS，把 app.js 按模块拆成 15 个小文件 + 引入 [Alpine.js](https://alpinejs.dev) 做轻量响应式 | 低 | 1 天 | ⭐⭐⭐ |
| B. 引入 [Vue 3 + Element Plus](https://element-plus.org/)（组件库自带 Cascader / Autocomplete / Table 等，采集箱抽屉几乎零写代码） | 中 | 3-5 天 | **⭐⭐⭐⭐⭐** |
| C. 全站重写 React + Ant Design | 高 | 2 周 | ⭐⭐ |

**产品强烈推荐 B**：Element Plus 的 `el-cascader`（级联）、`el-autocomplete`（可搜索下拉）、`el-drawer`（抽屉）、`el-table`（表格）都是本 PRD 需要的**开箱即用**组件，比自己造轮子快 10 倍且不容易崩。

### 4.3 迁移路径

1. **不动存量页面**：现有页面继续跑
2. **新页面用 Vue**：先把采集箱编辑抽屉用 Vue 3 + Element Plus 独立开发，通过 `<script type="module">` 挂到现有 HTML
3. **逐步替换**：新功能全部用 Vue，老页面按需重构，6 周内完成迁移
